# 기능요청 페이지 모바일 개선 — 설계서

- 작성일: 2026-06-11
- 대상 앱: `wedly-dev-request` (ERP 안에 iframe으로 임베드, 단독 주소로도 접근)
- 대상 파일: `app/page.tsx` (단일 클라이언트 컴포넌트, 642줄)
- 배경: 휴대폰에서 (1) 목록이 표 7칸이라 가로로 밀려 보기 불편, (2) 입력 '확인' 단계에서 같은 글이 미리보기+편집으로 두 번 쌓여 화면이 과도하게 길어짐.

## 목표

좁은 화면(휴대폰)에서 목록을 카드로, 입력 확인 단계를 한 칸으로 정리한다. **PC/태블릿 화면(표 레이아웃)은 변경하지 않는다.**

## 범위

**포함**
- 목록: 좁은 화면에서 표 대신 카드 스택 (스타일 A · 정보 한눈에)
- 입력: 확인 단계(STEP 3)에서 좁은 화면일 때 미리보기 숨기고 편집 한 칸만 표시
- 펼친 내용 렌더링 로직을 공용 조각으로 추출(표/카드 공유)
- 칩·드롭다운 터치 영역 확대(모바일 가독·조작성)

**제외 (그대로 둠)**
- PC 표 레이아웃, 모든 데이터, AI 구조화 흐름, 파일 첨부, 업데이트 팝업, 서버 API
- 입력 STEP 1(내용 작성)은 이미 모바일에서 잘 쌓임 → 변경 없음
- 전체화면 시트(옵션 B)는 채택하지 않음

## 분기 전략 (반응형)

- 기준 폭: Tailwind `md`(768px). `md` 미만 = 모바일(카드), `md` 이상 = 기존 표. (목록·입력 동일 분기점으로 통일. 9칸 표는 768px 미만에서도 좁아 `md`가 적절.)
- 구현: 표 블록과 카드 블록을 **둘 다 렌더**하고 CSS로 하나만 표시.
  - 표: `hidden md:block` (기존 `overflow-x-auto` 컨테이너에 추가)
  - 카드: `md:hidden`
- JS 뷰포트 측정 없이 CSS만 사용 → 로드 시 깜빡임 없음, iframe 안에서도 안정적. 요청 개수가 적어 마크업 이중 렌더 비용은 무시 가능.
- 상태 공유: 두 블록 모두 동일한 `requests` 배열과 동일한 펼침/삭제 상태(`expandedId`, `confirmDeleteId`)를 사용 → 동작 일관.

## 목록 카드 설계 (A · 정보 한눈에)

카드 1개 = 요청 1건. 외곽: `rounded-xl border border-wedly-bd` (내부 카드 표준), 카드 간 세로 간격.

구성(위→아래):
1. 윗줄: `#{no}`(왼쪽, muted) · **상태 칩**(오른쪽) — 칩 누르면 드롭다운(시작 전/대기중/진행 중/완료)
2. 제목: 굵게, `text-wedly-navy`, 최대 2줄(`line-clamp-2`)
3. 칩 줄: 앱 배지(`wedly-` 접두 제거) · **카테고리 칩** · **우선순위 칩** (각 칩 누르면 드롭다운으로 즉시 변경)
4. 아랫줄: `{requester} · {날짜}`(왼쪽) · `더보기 ›`/`접기`(오른쪽)
5. 카드 본문 또는 '더보기' 탭 → 그 자리에서 펼침:
   - 펼친 영역: 전체 내용(공용 렌더러) + **복사** 버튼 + **삭제**(삭제 확인 2단계: 삭제/취소)
   - 삭제는 펼친 영역 안에만 배치 → 목록에서 실수 탭 방지

편집 동작:
- 기존 `InlineSelect`(드롭다운) 재사용. 카드의 상태/카테고리/우선순위 칩에 그대로 연결(`handleUpdate(id, field, value)`).
- 터치 영역: 칩 및 드롭다운 항목 패딩을 키워 손가락 탭이 쉽게(모바일 최소 높이 확보).

## 입력 확인 단계 설계 (A · 한 칸 통합)

STEP 3(review)의 미리보기/편집 2열 그리드(`grid-cols-1 md:grid-cols-2`):
- 미리보기 칸(읽기 전용): `hidden md:block` 로 변경 → 좁은 화면에서 숨김
- 편집 칸(편집 가능 textarea): 좁은 화면에서 단독 전체폭으로 표시(높이 유지)
- 결과: 모바일에서 같은 글 중복 제거. 제목 입력·카테고리·우선순위 버튼 줄은 현행 유지(이미 줄바꿈됨).

> 분기점은 목록과 동일한 `md`(768px). 미리보기 칸에 `hidden md:block`을 더해 모바일에서 숨기고, 편집 칸은 1열에서 단독 전체폭으로 남는다.

## 코드 정리

- 펼친 내용의 글자 꾸미기 로직(현재 `app/page.tsx` 558~572줄의 `content.split("\n").map(...)`)을 작은 컴포넌트(예: `RequestBody`)로 추출.
- 표의 펼친 행과 카드의 펼친 영역이 이 컴포넌트를 공유 → 동일 표시, 중복 제거.
- 추출 외 불필요한 리팩토링은 하지 않음.

## 디자인 규칙

- 색상은 dev-request 앱 토큰만 사용(`bg-bg-blue`, `text-wedly-accent` 등). raw Tailwind 컬러 스케일 금지.
- 카드 외곽/상태 박스 팔레트는 ERP 디자인 시스템과 동일 원칙.

## 검증 계획

1. `npx next build` 성공.
2. 배포 후 실제 브라우저(폭 ~375px)로 확인:
   - 목록이 카드로 보임(표 가로 스크롤 없음)
   - 상태/카테고리/우선순위 칩 탭 → 드롭다운 → 값 변경 반영
   - 카드 탭 → 펼침 → 전체 내용 + 복사 + 삭제(2단계) 동작
   - '새 요청' → AI/바로 등록 → 확인 단계에서 편집 한 칸만 표시
   - PC 폭(≥768px)에서 기존 표·2열 미리보기 그대로(회귀 없음)
   - 숨은 오류(errors)·실패 통신(4xx/5xx) 0건, 캡처 증거
3. raw 컬러 grep 검증(브랜드 예외 외 0건).

## 엣지 케이스 / 위험

- 긴 제목: `line-clamp-2`로 2줄 제한, 펼치면 전체 표시.
- 빈 내용(content 없음): 펼침 영역에 "내용 없음" 또는 제목만. 기존 동작 유지.
- 드롭다운이 카드 경계 밖으로 나갈 때: 드롭다운은 절대배치이므로 카드가 잘리지 않게 부모 `overflow` 주의(카드 컨테이너에 `overflow-visible`).
- 삭제 확인 상태(`confirmDeleteId`)가 표/카드 양쪽에서 공유되므로 한쪽에서 연 확인이 다른 쪽과 충돌하지 않게 동일 id 기준 유지.
- iframe 높이(`calc(100vh-80px)`) 안에서 카드 스택 스크롤 정상 동작 확인.
